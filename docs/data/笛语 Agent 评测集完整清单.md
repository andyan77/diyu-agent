 # 笛语 Agent 评测集完整清单

> **版本:** v1.0  
> **日期:** 2026-02-20  
> **定位:** 本文档列出笛语 Agent 系统中所有需要评测集验证的概率性功能。  
> **核心原则:** 凡是依赖 LLM 判断的功能，代码测试只能验证"管道通不通"，评测集才能验证"判断对不对"。

---

## 总览

笛语系统共需 **28 个评测集**，分布在 5 个层级：

| 层级 | 评测集数量 | 覆盖范围 |
|------|-----------|---------|
| Brain 层（5大固有能力） | 12 | 意图/记忆/角色/调度/对话 |
| Knowledge 层 | 6 | 检索/召回/FK联动/继承 |
| Skill 层 | 5 | 内容生产/搭配/合规/平台适配 |
| 跨层闭环 | 3 | 上下文组装/降级/Promotion |
| 安全与对抗 | 2 | 注入防御/记忆投毒 |

---

## 一、Brain 层评测集（12 个）

### E-01 意图二分类

| 字段 | 内容 |
|------|------|
| **评测对象** | Brain 意图判断：纯对话 vs 需要"做事" |
| **架构锚点** | 00 Section 9 请求处理全流程 → 意图判断分叉 |
| **样本量目标** | ≥ 200 条 |
| **输入格式** | 用户单条消息（含上下文窗口摘要） |
| **标准答案** | `chat`（纯对话）或 `action`（需要调 Skill） |
| **核心指标** | 准确率 ≥ 95%；误判率（chat 判为 action）≤ 3%；漏判率（action 判为 chat）≤ 5% |
| **必须覆盖的case类型** | ①明确意图（"帮我写文案"）②口语化（"弄个东西发抖音"）③感慨非请求（"内容创作真难"）④否定（"不用帮我了"）⑤寒暄（"你好/早上好"）⑥模糊（"这件衣服怎么样"） |
| **谁出题** | 产品负责人（你）+ 一线用户访谈 |
| **跑测频率** | 每次 Prompt 修改后；每周定时回归 |
| **对应 Phase** | Phase 2（首条端到端闭环） |

### E-02 Skill 路由准确率

| 字段 | 内容 |
|------|------|
| **评测对象** | Skill Router：已判定"需要做事"后，路由到哪个 Skill |
| **架构锚点** | 03 Section 2 SkillProtocol.can_handle() + Skill Router |
| **样本量目标** | ≥ 150 条（每个已注册 Skill 至少 40 条） |
| **输入格式** | 用户消息 + 当前已注册 Skill 列表 |
| **标准答案** | 应路由到的 Skill name（如 `content_writer` / `merchandising`）或 `none`（无匹配 Skill） |
| **核心指标** | 路由准确率 ≥ 95%（B3-1 验收标准）；置信度灰区（0.4-0.6）的误判率单独统计 |
| **必须覆盖的case类型** | ①明确单 Skill（"帮我写小红书文案"→content_writer）②明确另一 Skill（"这件外套配什么裤子"→merchandising）③无匹配 Skill（"今天天气怎么样"→none）④复合意图（"搭好了写个文案"）⑤口语化（"随便搭搭"）⑥跨场景干扰（内容生产场景中提到搭配） |
| **谁出题** | 产品负责人 + 各 Skill 领域专家 |
| **跑测频率** | 每次新 Skill 注册后；每次 can_handle() 或 Router 逻辑修改后 |
| **对应 Phase** | Phase 3（知识与技能接入） |

### E-03 Session 记忆连贯性

| 字段 | 内容 |
|------|------|
| **评测对象** | 会话内上下文理解：指代消解、话题延续、指令继承 |
| **架构锚点** | 01 Brain 对话引擎 + Memory Core session scope |
| **样本量目标** | ≥ 50 组多轮对话（每组 4-8 轮） |
| **输入格式** | 多轮对话序列 + 每轮的验证问题 |
| **标准答案** | 每轮系统应表现出的理解（如"第3轮说'短一点'→修改上一轮生成的文案长度"） |
| **核心指标** | 上下文保持率 ≥ 90%（10轮内无上下文丢失）；指代消解准确率 ≥ 85% |
| **必须覆盖的case类型** | ①指代消解（"改成那个/换一下"）②话题切换后回溯（"还是刚才那个好"）③隐式指令继承（第1轮指定风格，第3轮不重复但仍生效）④长对话衰减（第8轮还记得第1轮的设定吗）⑤纠错（"不是那个，我说的是…"） |
| **谁出题** | 产品负责人 + QA 团队 |
| **跑测频率** | 每次 Context Assembler 或会话窗口逻辑修改后 |
| **对应 Phase** | Phase 2 |

### E-04 Personal 记忆提取质量

| 字段 | 内容 |
|------|------|
| **评测对象** | 记忆引擎 Observer：从对话中提取观察写入 Memory Core 的质量 |
| **架构锚点** | 01 Section 2 记忆引擎 Observer/Analyzer + 管线1 Memory 写入 |
| **样本量目标** | ≥ 100 段对话（每段含 1-3 个可提取观察） |
| **输入格式** | 一段完整对话 |
| **标准答案** | 应提取的观察列表 + 不应提取的内容列表 |
| **核心指标** | 提取召回率 ≥ 80%（该提的提到了）；提取精确率 ≥ 85%（不该提的没提）；误提率 ≤ 10% |
| **必须覆盖的case类型** | ①明确偏好（"我喜欢简约风"→应提取）②隐含偏好（连续选简约→应提取）③一次性情绪（"今天好烦"→不应提取为长期特征）④纠正行为（"不要花哨"→反向偏好）⑤临时指令 vs 长期偏好（"这次用正式语气" vs "我一般喜欢口语化"）⑥角色线索（"我们门店主要卖…"→门店经营者） |
| **谁出题** | 产品负责人 |
| **跑测频率** | 每次 Observer Prompt 修改后 |
| **对应 Phase** | Phase 2 |

### E-05 Personal 记忆应用准确性

| 字段 | 内容 |
|------|------|
| **评测对象** | 跨会话记忆使用：系统是否在新对话中正确引用已学到的偏好 |
| **架构锚点** | 01 Section 4.1 Context Assembler Step 1 + injection_receipt |
| **样本量目标** | ≥ 60 组跨会话测试（需预埋 memory_items） |
| **输入格式** | 预设的 memory_items + 新会话中的用户消息 |
| **标准答案** | 系统回复应体现的记忆引用（或不应引用的场景） |
| **核心指标** | 记忆引用准确率 ≥ 85%；过度引用率 ≤ 10%（不该用的记忆被用了）；记忆遗漏率 ≤ 15% |
| **必须覆盖的case类型** | ①"像上次那样写"→正确使用历史偏好 ②"这次不同，正式一点"→临时覆盖不污染长期偏好 ③无明确指令→默认使用长期偏好 ④矛盾记忆→使用最新的 ⑤相关但不精确匹配的记忆→不生搬硬套 |
| **谁出题** | 产品负责人 |
| **跑测频率** | 每次 Reranking 参数或注入策略修改后 |
| **对应 Phase** | Phase 2（基础）→ Phase 3（完善） |

### E-06 角色适配

| 字段 | 内容 |

| **架构锚点** | 01 Brain 角色适配 + 00 Section 3 三大场景的分层管控 |

| **核心指标** | 角色区分度 ≥ 80%（评审者能从回复中正确判断用户层级）；不越权率 100%（门店用户不应看到总部策略） |
| **必须覆盖的case类型** | ①brand_hq 管理员→专业/策略级回复 ②regional_agent→执行建议+区域化 ③franchise_store→简洁实操+口语化 ④不同 role 同 tier（admin vs viewer）⑤降级场景（Knowledge Stores 不可用时角色适配退化） |
| **谁出题** | 产品负责人 + 品牌方代表 |

  ## 七、CI 建议执行规则（可直接落地）

  1. PR Gate：只跑受影响评测集 + 高风险常驻集（E-16,E-20,E-27,E-31,E-33）。
  2. Nightly Gate：跑全量 E-01~E-33，输出层级看板。
  3. Release Gate：四层 gate 全绿才允许发布。
  4. Blocker 优先级：安全/权限/合规 > 数据契约 > 体验质量。
